# OMEGA_BOT_Iulik_Skornykova
# Модуль видеотрансляции и компьютерного зрения
Данное приложение является клиентской частью системы управления, обеспечивающей прием видеопотока в реальном времени, детектирование объектов и фоновую запись логов.

## 1. Технологический стек видеопотока
Для работы с видео используются следующие компоненты:

- OpenCV (4.x+): Основная библиотека обработки изображений.Требуется сборка с флагами:
  - WITH_GSTREAMER=ON — для захвата и декодирования UDP/RTP потоков.
  - BUILD_opencv_dnn=ON — для запуска нейросети YOLO11 (модуль Deep Neural Network).
    
- GStreamer: Используется в качестве бэкенда для захвата UDP-потока.
  
- YOLO11 (ONNX): Глубокое обучение для детектирования объектов (в частности, мяча).
  
- Qt6/Qt5: Визуализация кадров в графическом интерфейсе.

## 2. Захват и передача видеопотока на сервере 
1. Захват и подготовка
- Для работы с физической камерой и формирования потока используются:
- Интерфейс подключения: V4L2 (Video4Linux2). Используется для прямого взаимодействия с USB-камерами (/dev/video0).
- Формат захвата: Motion JPEG (MJPEG). Камера сразу отдает сжатые кадры, что минимизирует нагрузку на CPU сервера (например, Raspberry Pi или Jetson).
- Разрешение и FPS: Поток настроен на 1280x720 (720p) при 30 кадрах в секунду для баланса между детализацией и скоростью обработки.
  
2. Сетевые настройки
  
- Протокол: UDP (User Datagram Protocol). Выбран из-за отсутствия повторных запросов потерянных пакетов, что критично для видео реального времени.
- Прикладной протокол: RTP (Real-time Transport Protocol). Выполняет роль «контейнера»: нумерует пакеты и добавляет временные метки, чтобы клиент мог собрать кадр без искажений.
- Адресация: * IP Клиента: В команде запуска сервера необходимо указать актуальный IP-адрес принимающей стороны.
- Порт: Используется стандартный порт 5000.

## 3. Обработка видеопотока на стороне клиента 

1. Прием (UDP/GStreamer):
- Захват RTP-потока через udpsrc (порт 5000).
- Декодирование JPEG в формат cv::Mat.
  
2. Анализ 
- Детектирование объектов моделью YOLO11 через модуль cv::dnn.
  
3. Логирование 
- Фоновая запись в .avi через отдельный поток VideoBufferLogger (сохраняет каждый 3-й кадр для оптимизации ресурсов).
  
4. Передача в клиентсоке Qt-приложение с результатами анализа 
- Отрисовка Bounding Boxes и меток классов.
- Конвертация BGR → RGB и масштабирование под размер окна Qt.
- Передача в клиентсоке Qt-приложение.
  
## 4. Инструкция по развертыванию
1. Убедиться, что в систему предустановлена библиотека OpenCv с поддержкой GStreamer и модулем dnn

2. Подготовьте модель: Разместите файл yolo11n.onnx в корневой папке с исполняемым файлом

3. Запустите трансляцию на стороне сервера (например, с Raspberry Pi или другой камеры).Убедитесь, что поток направлен на IP вашего клиента и порт 5000.

## Конфигурация и кастомизация
1. Смена модели и объекта детектирования
В методе updateVideoFrame (файл client.cpp) можно изменить логику работы нейросети:
- Путь к модели:
  std::string modelPath = "yolo11n.onnx"; // Укажите путь к вашей модели в формате .onnx
  Выбор объекта (ID класса):
- Детектируемый объект выбирается по ID в стандартном наборе COCO (YOLO), индекс 32 соответствует мячу.
if (classIds[idx] == 32) { // Замените 32 на нужный ID (например, 0 для человека)
    cv::putText(frame, "BALL", ...); // Замените текст метки
}

2. Настройка сетевой передачи данных
Если видео передается с другого устройства или по другому порту, измените строку инициализации pipeline:
- Порт: port=5000 — должен совпадать с портом отправителя.
- Кодек: Если сервер отправляет не JPEG, а, например, H264, пайплайн нужно заменить на соответствующий (rtph264depay ! avdec_h264).
  
3. Параметры записи логов
В классе VideoBufferLogger и конструкторе MainWindow можно настроить:
- Директория сохранения: std::string log_dir = "/home/user/Desktop/video_log";
- Частота записи: Переменная LOG_EVERY_N = 3 (записывать каждый 3-й кадр).
- Разрешение видео: TARGET_SAVE_SIZE(320, 240) (влияет на размер файла лога).
